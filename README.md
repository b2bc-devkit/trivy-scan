# trivy-scan

Run [Aqua Security's Trivy](https://github.com/aquasecurity/trivy) vulnerability scanner via `npx` on Linux, macOS and Windows — **no Docker, no Homebrew, no manual installation**.

```bash
npx @b2bc-devkit/trivy-scan image python:3.9
```

On first use the wrapper downloads the official Trivy binary for your OS/CPU from GitHub Releases, **verifies its SHA-256 checksum**, caches it in your user cache directory and runs it. Every subsequent run starts in milliseconds with zero network traffic and zero wrapper output.

## Usage

`trivy-scan` is a fully transparent wrapper: every argument after the command name is forwarded to the real `trivy` binary **byte-for-byte** (`process.argv.slice(2)` → `spawn(trivy, args, { stdio: "inherit" })`). No argument parsing, no injected flags, no output rewriting. Anything the [Trivy docs](https://trivy.dev/latest/docs/) describe works as-is:

```bash
# Scan a container image
npx @b2bc-devkit/trivy-scan image python:3.9

# Scan the current directory, only HIGH/CRITICAL findings
npx @b2bc-devkit/trivy-scan fs . --severity HIGH,CRITICAL

# Scan a remote git repository, JSON report to a file
npx @b2bc-devkit/trivy-scan repo https://github.com/example/repo -f json > result.json

# Fail the build when vulnerabilities are found (exit code passes through 1:1)
npx @b2bc-devkit/trivy-scan fs . --exit-code 1 --severity CRITICAL

# IaC / secrets / licenses — any trivy subcommand works
npx @b2bc-devkit/trivy-scan config ./terraform
npx @b2bc-devkit/trivy-scan fs --scanners secret .
```

Colors, progress bars, interactive output, stdout/stderr separation and exit codes are preserved 1:1, so shell pipelines and CI gates behave exactly as with a native Trivy installation.

## How it works

```
npx @b2bc-devkit/trivy-scan <args>
      │
      ▼
1. Resolve platform     process.platform/arch → official artifact name
                        (darwin→macOS, linux→Linux, win32→windows; x64→64bit, arm64→ARM64)
2. Resolve version      latest release via one HTTPS redirect (24h cached),
                        falling back to the package-pinned release when offline
3. Cache hit?     yes → run immediately (zero network, zero wrapper output)
      │ no
      ▼
4. Download             https://github.com/aquasecurity/trivy/releases/download/…
5. Verify SHA-256       against a trusted checksum — mismatch = hard abort,
                        the file is discarded and never executed
6. Extract + chmod +x   tar.gz (Linux/macOS) or zip (Windows), atomic rename
      │
      ▼
   spawn(trivy, args, { stdio: "inherit" }) → exit code passes through
```

## Supported platforms

| OS      | Architectures                     | Artifact                          |
| ------- | --------------------------------- | --------------------------------- |
| Linux   | x64, arm64, arm, ppc64, s390x     | `trivy_<v>_Linux-*.tar.gz`        |
| macOS   | x64 (Intel), arm64 (Apple Si)     | `trivy_<v>_macOS-*.tar.gz`        |
| Windows | x64, arm64 (via x64 emulation)    | `trivy_<v>_windows-64bit.zip`     |

Unsupported combinations abort with a clear error before anything is downloaded.

## Version control

By default the wrapper tracks the **latest** stable Trivy release, checking at most once per 24 h through the `releases/latest` HTTP redirect — the GitHub REST API is never used, so **API rate limits cannot break your CI**. Override via environment variable:

| `TRIVY_SCAN_VERSION` | Behavior                                                                  |
| -------------------- | ------------------------------------------------------------------------- |
| `latest` *(default)* | Newest upstream release; falls back to the pinned release when offline    |
| `pinned`             | The release pinned inside this package — deterministic and offline-safe   |
| `v0.71.0` / `0.71.0` | Exactly that release                                                      |

```bash
# Reproducible CI runs, zero version-check traffic:
TRIVY_SCAN_VERSION=pinned npx @b2bc-devkit/trivy-scan fs . --exit-code 1
```

## Security model

1. **HTTPS-only, official source.** Artifacts come exclusively from `github.com/aquasecurity/trivy/releases`; redirects to any non-HTTPS location are refused.
2. **Package-pinned checksums.** The package ships hardcoded SHA-256 checksums for every artifact of the pinned release (see `src/config/PinnedRelease.ts`, regenerated by maintainers with `npm run update-pinned`). For the pinned version, verification requires **no trust in any live endpoint**.
3. **Release-manifest checksums.** For other versions (`latest`/explicit), checksums come from that release's official `trivy_<v>_checksums.txt`, fetched over TLS and cached beside the binary.
4. **Verify before extract.** The SHA-256 of every downloaded archive is compared against the trusted checksum **before extraction**. On mismatch the wrapper aborts with a `SECURITY` error, deletes the file and exits 1 — nothing is ever executed.
5. **Atomic installs.** Downloads and extraction use unique temp files finalized by `rename()`, so crashes or concurrent runs can never leave a corrupt binary in the cache.

For maximum supply-chain rigor use `TRIVY_SCAN_VERSION=pinned` and pin this package's own version in your lockfile: binary provenance is then fixed at `npm install` time.

## Caching

| OS      | Default cache location                                     |
| ------- | ---------------------------------------------------------- |
| Linux   | `$XDG_CACHE_HOME/trivy-scan` or `~/.cache/trivy-scan`       |
| macOS   | `~/Library/Caches/trivy-scan`                               |
| Windows | `%LOCALAPPDATA%\trivy-scan\Cache`                           |

Each version lives in its own subdirectory (`v0.72.0/trivy`). Override the root with `TRIVY_SCAN_CACHE_DIR`; delete the directory at any time to force a fresh, re-verified download. (This cache holds the Trivy *binary* — Trivy's own vulnerability DB cache is managed by Trivy itself, see `trivy clean`.)

## CI/CD

### GitHub Actions

```yaml
jobs:
  security-scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
      - name: Cache Trivy binary
        uses: actions/cache@v4
        with:
          path: ~/.cache/trivy-scan
          key: trivy-scan-${{ runner.os }}
      - name: Scan filesystem
        run: npx --yes @b2bc-devkit/trivy-scan fs . --exit-code 1 --severity HIGH,CRITICAL
        env:
          TRIVY_SCAN_VERSION: pinned   # deterministic runs
```

### GitLab CI

```yaml
security-scan:
  image: node:22
  variables:
    TRIVY_SCAN_VERSION: pinned
    TRIVY_SCAN_CACHE_DIR: .trivy-scan-cache
  cache:
    key: trivy-scan
    paths: [.trivy-scan-cache]
  script:
    - npx --yes @b2bc-devkit/trivy-scan fs . --exit-code 1 --severity HIGH,CRITICAL
```

## Exit codes

- Trivy's exit code is propagated unchanged (including `--exit-code` gates).
- Wrapper-level failures (unsupported platform, download/network error, checksum mismatch) exit `1` with a `trivy-scan: error: …` message on **stderr** — stdout always belongs to Trivy alone.

## Development

```bash
npm install            # installs deps and builds (prepare hook)
npm test               # unit tests (offline, fast)
npm run test:integration  # downloads real release artifacts, verifies checksums,
                          # extracts binaries and runs `trivy --version`
npm run update-pinned  # maintainers: re-pin to the latest (or given) release,
                       #   e.g. npm run update-pinned -- v0.73.0
```

The codebase is strict-mode TypeScript with a class-per-file OOP design: value objects (`TrivyVersion`, `PlatformTarget`, `Sha256Digest`), interfaces with constructor-injected implementations (`HttpClient`, `ChecksumProvider`, `ArchiveExtractor`, `VersionResolver`), a `TrivyInstaller` orchestrator, a `TrivyRunner` process host and a `CliApplication` composition root. Runtime dependencies are just `tar` and `unzipper`; everything else uses Node's native `https`, `crypto`, `fs/promises` and `child_process`.
